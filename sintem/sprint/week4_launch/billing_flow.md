# SINTEM. Биллинг-флоу и compliance

Стек: Telegram Bot API → Cloudflare Worker → NowPayments API (primary), CryptoCloud (fallback) → D1 для состояний подписки. Расчётная валюта в фактуре — рубли, расчёт on-chain — USDT TRC-20 / BTC / TON по выбору юзера.

## 1. Архитектура биллинг-объектов в D1

Таблицы:

```
users (telegram_id PK, created_at, plan, status, plan_expires_at, early_bird, lang)
invoices (id PK, user_id FK, provider, provider_invoice_id, amount_rub, amount_crypto, currency, status, created_at, paid_at, expires_at)
subscriptions (id PK, user_id FK, plan, started_at, expires_at, autorenew, source_invoice_id)
events (id PK, user_id, type, payload_json, ts) — для аудита и аналитики
quota_usage (user_id, agent, period_start, used) — счётчик запросов
```

Статусы `invoices.status`: `created → waiting → confirming → paid | expired | failed | refunded`.
Статусы `users.status`: `trial → active → grace → expired → churned`.

## 2. End-to-end флоу подписки

Сценарий: активная подписка истекает через 3 дня, Аслан-бот шлёт напоминание.

```
1. Cron Worker (каждые 6 часов) сканит users.plan_expires_at < now()+3d AND status=active
   ↓
2. В TG юзеру отправляется сообщение «Подписка истекает 09.05. Продлить?» с кнопкой [Продлить]
   ↓
3. Юзер жмёт inline-кнопку, callback_query летит на Worker /tg/callback
   ↓
4. Worker: 
   - читает users.plan, считает сумму (учитывает early_bird, annual-флаг)
   - вызывает POST https://api.nowpayments.io/v1/invoice
       { price_amount: 3900, price_currency: "RUB", 
         pay_currency: "USDTTRC20", 
         order_id: "<uuid>", 
         ipn_callback_url: "https://api.sintem.bot/np/webhook",
         success_url: "https://t.me/sintem_bot?start=paid_<uuid>",
         cancel_url: "https://t.me/sintem_bot?start=cancelled_<uuid>" }
   - получает {id, invoice_url, pay_address, pay_amount}
   - вставляет строку в invoices со status=created
   - отправляет юзеру в TG: «Оплатить ₽3900 ($42.4) → [Открыть страницу оплаты]» (URL invoice_url)
   ↓
5. Юзер открывает invoice_url в браузере → выбирает USDT/BTC/TON → платит со своего кошелька
   ↓
6. NowPayments вебхук → POST https://api.sintem.bot/np/webhook с HMAC SHA-512
   payload: { payment_status: "finished", order_id, actually_paid, ... }
   ↓
7. Worker /np/webhook:
   - проверяет HMAC через NOWPAYMENTS_IPN_SECRET (env binding)
   - находит invoice по order_id
   - если status в (waiting,confirming): обновляет на paid, ставит paid_at
   - создаёт строку в subscriptions (started_at=now, expires_at=now+30d или +365d)
   - users.plan = новый/прежний, users.status=active, users.plan_expires_at=expires_at
   - сбрасывает quota_usage за период
   - шлёт юзеру в TG: «Подписка активна до DD.MM.YYYY. Спасибо.»
   ↓
8. Если status=partially_paid (приехало меньше из-за курсового проскальзывания на >2%):
   - оставляем invoice waiting
   - шлём юзеру: «Получено $40.1 из $42.4. Доплатите $2.3 на тот же адрес в течение 24ч.»
```

Ровно тот же флоу для CryptoCloud в роли fallback: ендпойнт `https://api.cryptocloud.plus/v2/invoice/create`, вебхук на `/cc/webhook`, проверка по `secret_key` через `Authorization: Bearer`.

## 3. Маршрутизация NowPayments vs CryptoCloud

NowPayments — primary, потому что у него шире выбор валют (USDT TRC-20, BTC, ETH, TON, BNB) и он принимает РФ-карты на стороне их custodial flow для тех юзеров, у кого нет крипты вообще. Минус: периодически режут русские IP в личном кабинете (не у клиентов, а у мерчанта), поэтому управление аккаунтом через VPN.

CryptoCloud — fallback. Включается, если:
- NowPayments вернул 5xx 3 раза подряд за 30 секунд (circuit breaker в Worker), либо
- CryptoCloud дешевле для конкретной валюты, либо
- ручной флаг в users.preferred_provider.

Решение делает сам Worker: проверяет `KV: provider_health` (обновляется фоновым cron каждые 5 мин), и если NowPayments=down — выписывает invoice через CryptoCloud, юзер не замечает разницы.

## 4. Failed / expired платежи и grace period

Состояния:

- **invoice.expired**: NowPayments invoice живёт по умолчанию 1 час. Если юзер не заплатил — статус становится `expired`, бот шлёт «Не успели оплатить, попробуйте ещё раз» и предлагает создать новый invoice. Никаких санкций к подписке — старая ещё активна.
- **subscription.expires_at прошёл, оплаты нет** → users.status=`grace`, `plan_expires_at` остаётся прежним (для аналитики), `grace_until = expires_at + 3d`. В этом окне юзер видит мягкое сообщение «Подписка просрочена на N дней, продлите — иначе квота обнулится 09.05» и сохраняет полный доступ.
- **grace_until прошёл, оплаты нет** → users.status=`expired`, plan переходит в `free` (запросы блокируются, доступна только команда `/subscribe` и история на read-only за последние 30 дней).
- **30 дней в expired** → users.status=`churned`, флаг для win-back-кампаний (см. retention_playbook). История остаётся 90 дней, потом архивируется.

Cron-схема: каждые 6 часов Worker делает sweep, переводит state-машину пользователей. Все переходы пишутся в `events` для аудита.

## 5. Refund-политика

Официально: **refund нет**, потому что крипто-чарджбэк невозможен на стороне эквайринга и любой возврат — это ручная транзакция Аслана.

Исключение для первых 50 юзеров (early bird): **7-day money-back guarantee** через ручной процесс.

Процесс:
1. Юзер пишет в `/support` в боте «прошу возврат», прикладывает причину.
2. Аслан в течение 48 часов проверяет: подписка активна <7 дней, использовано <50% квоты, нет abuse-флагов.
3. Если ОК — Аслан вручную отправляет USDT TRC-20 на адрес, указанный юзером, в сумме `paid_amount × 0.97` (комиссия сети).
4. Subscription помечается status=`refunded`, доступ блокируется немедленно, early-bird сгорает.
5. Refund доступен один раз на user_id навсегда. Повторный — отказ.

После первых 50 юзеров пункт убирается из публичной оферты, остаётся только «hardship clause» — возврат на усмотрение Аслана при технической невозможности оказать услугу с нашей стороны (например, Groq отвалился на 7+ дней).

## 6. Crypto-volatility hedge

Проблема: между выпуском invoice (₽3900 = $42.4 при курсе 92 ₽/$) и фактической оплатой может пройти 30–60 минут, за которые BTC/USDT курс к рублю прыгает на 1–3%.

Что мы делаем:

- **Цена фиксируется в RUB**, NowPayments сам конвертит в крипту по курсу момента генерации invoice. Юзер видит сумму в крипте «pay_amount», которая live на 60 минут.
- На стороне Worker мы доверяем `actually_paid`, который NowPayments пересчитывает обратно в RUB по курсу момента поступления и сравнивает с ожиданием.
- Если `actually_paid >= price_amount × 0.98` (т.е. недополучили <2%) — засчитываем как полную оплату. 2% — это допустимый дрейф, дешевле чем гонять юзера за доплатой и терять конверсию.
- Если недополучили 2–10% — статус `partially_paid`, юзеру шлём «доплатите $X в течение 24ч на тот же адрес». NowPayments агрегирует поступления.
- Если недополучили >10% — invoice автоматически закрывается, средства возвращаются на адрес отправителя через NowPayments (стандартное поведение их custody).
- USDT/RUB риск держим на стороне NowPayments, мы получаем из них уже либо USDT на наш hot-wallet, либо RUB на расчётный счёт (когда оформим ИП).

Хедж в одну сторону (юзер недоплатил) — поглощаем 2% дрейф. Хедж в другую сторону (переплатил) — оставляем переплату на счёте, юзеру в TG приходит «у вас баланс +$0.8, спишется при следующем продлении».

## 7. Compliance, налоги, юр-форма Аслана

### Что мы продаём с точки зрения НК РФ

Услуга = «предоставление доступа к программно-аппаратному комплексу для автоматизированной обработки текстовых данных». В классификаторе ОКВЭД 2 — **62.09** (деятельность, связанная с использованием вычислительной техники и информационных технологий, прочая) и/или **63.11.1** (деятельность по обработке данных). Это **не** «образовательные услуги», **не** «консультации» и **не** «реклама», их формулировок избегаем в оферте.

Реализация: «Реализация контентных услуг в электронной форме» по 174-ФЗ для целей НДС не применима, потому что мы на УСН.

### Юр-форма: ИП или самозанятый

Самозанятый (НПД, 422-ФЗ):
- Лимит выручки 2.4 млн ₽/год. На 10 платящих по ₽3900 = ₽468К/год — укладывается.
- На 200 платящих × ₽3900 = ₽9.36М/год — **уже не укладывается**, лимит пробивается на 50-м платящем.
- Нельзя нанимать сотрудников по трудовому договору.
- Нельзя принимать оплату от юрлиц на счета юрлиц с НДС.
- Чек выдаётся через «Мой налог», но **только за фактический рублёвый приход**. Крипто-приход самозанятый легально не принимает (нет нормы в 422-ФЗ для криптоактивов как объекта услуги).

ИП на УСН-доходы (6%):
- Лимит 200 млн ₽/год — с большим запасом.
- Может работать через расчётный счёт банка, который примет рублёвый платёж от NowPayments после конвертации.
- ОКВЭД 62.09 + 63.11.1 при регистрации.
- Налог 6% от поступлений + страховые взносы (~₽57К/год фикс + 1% с дохода >₽300К).
- Книга учёта доходов (КУДиР), сдача декларации раз в год.

**Рекомендация для Аслана: ИП на УСН-доходы (6%) до старта продаж.**

Причины:
1. Криптоприход через NowPayments легально оформляется только если у вас есть юр.форма, способная принять рублёвую выплату с расчётного счёта payment-провайдера. NowPayments перечисляет на ИП-счёт после KYC. Самозанятому они платить не будут (нет реквизитов расчётного счёта).
2. На 50-м платящем юзере лимит самозанятого пробьётся, и придётся срочно переоформляться, теряя 1–2 недели биллинга.
3. ИП регистрируется за 3 рабочих дня через Госуслуги, госпошлина 0 ₽ при электронной подаче, выездов в ФНС нет.
4. Расчётный счёт в Точке/Тинькофф открывается за 1 день, тарифы ИП-старт — бесплатно при оборотах <₽500К/мес.

### Что нужно сделать ДО первого платного юзера

1. Подать на регистрацию ИП на УСН-доходы (6%) с ОКВЭД 62.09, 63.11.1, 73.20 (резерв под маркетинг-активности).
2. Открыть расчётный счёт.
3. Пройти KYC в NowPayments (паспорт, селфи, ИНН, ОГРНИП). Это 3–5 рабочих дней.
4. Установить онлайн-кассу (54-ФЗ): для услуг через интернет нужен облачный ОФД (например, АТОЛ-онлайн). Чек пробивается на каждый платёж, отправляется в TG юзеру и на email если есть.
5. Сформировать публичную оферту с пунктами: предмет (доступ к боту), цена (рубли), срок (месяц/год), порядок расчётов (через payment-провайдера), refund (только в случае hardship), персональные данные (152-ФЗ). Оферта вывешивается по статичному URL, в боте — ссылка плюс чекбокс «Согласен» при первой оплате.
6. Зарегистрироваться в Роскомнадзоре как оператор персональных данных (онлайн через [pd.rkn.gov.ru](https://pd.rkn.gov.ru), бесплатно, ~14 дней).
7. ОРИ-реестр **не нужен** — мы Telegram-бот, а не отдельный мессенджер. Бот ходит по чужим API, реестр организаторов распространения информации к нам не применим, пока у нас нет собственного канала обмена сообщениями между пользователями.

### 152-ФЗ

Хранимые персданные: telegram_id, username, lang, история запросов. По 152-ФЗ это «иные данные» (не специальные). Юзер при первом запуске бота получает уведомление: «Используя бот, вы соглашаетесь на обработку telegram_id для целей оказания услуги. Согласие можно отозвать командой /forget». При отзыве — soft-delete: telegram_id заменяется на хеш, история обнуляется через 30 дней (срок хранения для бухгалтерии).

### Что НЕ оформляем

- ВЭД-договор с NowPayments не нужен: они юр.лицо в Эстонии, для российского ИП это не валютный контроль, а оплата услуг payment-провайдера, проходит как поступление от нерезидента в ₽ через банк-корреспондент. ВЭД-паспорт сделки не требуется при сумме <₽600К на одну транзакцию.
- Лицензия на криптодеятельность — мы не криптообменник, мы получатель платежа. 259-ФЗ «О цифровых финансовых активах» к нам не применим.

## 8. Чеки и фискализация

Каждый успешный paid-вебхук:
1. Worker вызывает API АТОЛ-онлайн POST `/possystem/v4/<group_code>/sell` с реквизитами чека (наименование «SINTEM подписка Рост, 1 мес», цена 3900, СНО 1=УСН-доходы, признак способа расчёта 4=полный расчёт, признак предмета расчёта 4=услуга).
2. Получает `uuid` чека, ждёт callback от ОФД (1–60 сек).
3. URL чека из ответа сохраняем в `invoices.fiscal_url`, отправляем юзеру в TG.

Если фискализация упала — invoice не считается закрытым с точки зрения compliance, но услуга юзеру оказывается (плохой UX иначе). Worker ставит флаг `fiscal_pending=true`, через cron каждые 30 минут ретраит до успеха или 24ч лимита, после чего эскалирует Аслану в `/admin`-чат.

## 9. Что хранится в env Cloudflare Worker

Никаких секретов в коде. В Worker bindings:
- `NOWPAYMENTS_API_KEY` (secret)
- `NOWPAYMENTS_IPN_SECRET` (secret)
- `CRYPTOCLOUD_API_KEY` (secret)
- `CRYPTOCLOUD_SECRET` (secret)
- `TG_BOT_TOKEN` (secret)
- `ATOL_LOGIN`, `ATOL_PASSWORD`, `ATOL_GROUP_CODE` (secret)
- `ADMIN_CHAT_ID` (var)

Ротация секретов раз в 90 дней через `wrangler secret put`.

## 10. Метрики, которые надо снимать с биллинга

- invoice → paid conversion rate (целевой ≥75%, ниже — индикатор UX-проблемы в checkout).
- Средний time-to-pay (создан → paid). Цель ≤ 12 минут.
- Доля partially_paid и доля expired (норма <5% и <15% соответственно).
- Распределение валют оплаты (USDT vs BTC vs TON) — для понимания, какие сети держать «хоткими».
- Drop-off на webhook handshake (пишем в events.type=`webhook_failed`, цель <0.5%).
